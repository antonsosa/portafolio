Laboratorios IT / SOC

Objetivo:
Diseñar y configurar un entorno virtualizado para prácticas técnicas orientadas a infraestructura IT y simulaciones básicas de seguridad ofensiva/defensiva. Este laboratorio sirve como base para pruebas de escaneo, automatización, gestión de políticas y simulación de redes corporativas.

Herramientas utilizadas:
VirtualBox, Windows Server 2022, Windows 10, PowerShell.

  • Creación de VM en VirtualBox

    Se definió el nombre, carpeta de almacenamiento, ISO de Windows Server 2022 y tipo de sistema operativo. Se optó por instalación manual para mayor control.


    Se asignaron 3072 MB de RAM, 2 núcleos de CPU y 50 GB de disco, suficientes para prácticas de dominio y servicios básicos.


  • Configuración de red interna en VirtualBox

    Se habilitó el adaptador de red y se configuró como Red interna con el nombre RedLabNueva. Esta configuración permite la comunicación entre las VMs sin acceso externo, simulando una red corporativa aislada.
    ACTUALIZACIÓN - Esta corrección fue clave para resolver problemas posteriores de resolución DNS y conectividad entre el servidor y los clientes.


  • Instalación de Windows Server 2022

    • La VM inicia correctamente desde la ISO oficial de evaluación en español.


    • Se seleccionó Windows Server 2022 Standard con GUI para facilitar la administración visual y configuración de roles.


    • Se creó un volumen principal de 50 GB, suficiente para prácticas de AD, GPOs y servicios básicos.


    • Instalación en progreso sin errores, confirmando compatibilidad del entorno virtual.


    • Primera pantalla de inicio tras instalación. Se procede a configurar el sistema.

Resultado y aprendizaje obtenido:
Aprendí a montar y trabajar en un entorno virtualizado completo de Windows Server y un cliente de Windows 10 para prácticas técnicas.

Objetivo:
Implementar y validar un entorno de Active Directory desde cero en Windows Server 2022, simulando una infraestructura corporativa con control de identidad, autenticación centralizada y estructura organizativa. Este laboratorio establece la base para la gestión de usuarios, aplicación de políticas (GPOs), y prácticas de seguridad defensiva en red local.

Herramientas utilizadas:
VirtualBox, Windows Server 2022, Windows 10, Active Directory Users and Computers (ADUC), PowerShell, CMD.

  • Instalación del rol AD DS

    Desde el Administrador del servidor, se confirma que el rol Active Directory Domain Services está instalado y operativo en el servidor WIN-A40J1L1PU1T con IP 192.168.10.10. Esta configuración es esencial para la gestión de identidad y acceso en el entorno de laboratorio.


  • Configuración del dominio Active Directory

    Se creó el dominio empresa.local con nivel funcional Windows Server 2016, compatible con políticas avanzadas y servicios de replicación. La descripción técnica indica que el entorno está diseñado para prácticas de administración y seguridad.


  • Estructura organizativa en Active Directory

    Se creó una estructura personalizada en Active Directory con unidades organizativas (OU) como Usuarios, Equipos y TI. Dentro de estas se gestionan usuarios y grupos de seguridad, simulando una red corporativa segmentada por funciones.


  • Unión del cliente Windows 10 al dominio

    El equipo DESKTOP-UHGP23N fue unido exitosamente al dominio empresa.local. Esta unión permite la autenticación centralizada, aplicación de políticas de grupo (GPOs) y validación de conectividad con el controlador de dominio.


  • Validación de identidad y conectividad en el dominio

    Desde el cliente DESKTOP-UHGP23N, se ejecutaron los comandos whoami y ping para validar la autenticación y conectividad con el controlador de dominio 192.168.10.10. El resultado confirma que el usuario está correctamente vinculado al dominio empresa.local y que la red interna está operativa.


Resultado y aprendizaje obtenido:
Aprendí a implementar y validar un entorno de Active Directory desde cero, configurando el rol AD DS, creando un dominio funcional y estructurando unidades organizativas con usuarios y grupos personalizados. Comprendí el proceso de unión de clientes al dominio, verifiqué la autenticación centralizada y confirmé la conectividad entre equipos. Este laboratorio fortaleció mi comprensión sobre la gestión de identidad, control de acceso y preparación de entornos corporativos para futuras políticas y automatización.

Objetivo:
Configurar y validar los servicios DHCP y DNS en Windows Server 2022 para automatizar la asignación de direcciones IP y garantizar la resolución de nombres dentro del dominio empresa.local.

Herramientas utilizadas:
VirtualBox, Windows Server 2022, Windows 10, Consola DHCP, Consola DNS, CMD, PowerShell

  • Instalación del rol DHCP en Windows Server

    Se instaló el rol DHCP Server en el equipo WIN-A2011HP1MTI con IP 192.168.1.60. El servicio está activo y genera eventos informativos, lo que indica que está listo para gestionar la asignación dinámica de direcciones IP en la red interna.


  • Configuración del ámbito DHCP

    Se configuró el ámbito RedInterna en el servidor DHCP, definiendo el rango de direcciones 192.168.10.100 a 192.168.10.200 con una duración de concesión de 8 días. Esta configuración permite la asignación automática de IPs a los clientes dentro de la red empresa.local.


  • Asignación automática de IP al cliente

    Desde el cliente DESKTOP-UHGP23N, se ejecutaron los comandos ipconfig /flushdns y ipconfig /renew para solicitar una nueva dirección IP al servidor DHCP. El resultado muestra una IP dinámica dentro del rango 192.168.18.100–200 (según ámbito configurado), confirmando que el servicio DHCP está funcionando correctamente y que la configuración DNS fue aplicada.


  • Instalación del rol DNS en Windows Server

    El rol DNS Server fue instalado junto con AD DS en el servidor WIN-A0011PU1HTT. El servicio está activo y genera eventos de advertencia iniciales (ID 4013), típicos durante la fase de configuración. Este rol permite la resolución de nombres dentro del dominio empresa.local, facilitando la conectividad entre clientes y servidores.


  • Validación de resolución DNS con nslookup

    Desde el cliente DESKTOP-UHGP23N, se ejecutó el comando nslookup empresa.local para validar la resolución de nombres en el dominio. Aunque el servidor DNS aparece como Unknown debido a la falta de zona inversa, el nombre empresa.local se resolvió correctamente a la IP 192.168.10.10, confirmando que el servicio DNS está operativo.


  • Registro dinámico del cliente en DNS

    Se validó que el cliente DESKTOP-UJ6P3AN fue registrado automáticamente en la zona win-adjapi.lujait del servidor DNS. El registro tipo A contiene la IP 192.168.10.12 y el nombre completo DESKTOP-UJ6P3AN.win-adjapi.lujait, confirmando que los servicios DHCP y DNS están integrados correctamente.


Resultado y aprendizaje obtenido:
Aprendí a configurar y validar los roles de DHCP y DNS en un entorno Windows Server, asegurando la asignación automática de direcciones IP y la resolución de nombres dentro del dominio. Implementé un ámbito DHCP con parámetros personalizados, incluyendo puerta de enlace, DNS y duración de concesión. Verifiqué la correcta resolución de nombres mediante pruebas de conectividad y comandos como nslookup y ping. Este laboratorio fortaleció mi comprensión sobre la infraestructura de red. Comprendí la interacción entre servicios como DHCP, DNS y Active Directory.

Objetivo:
Aplicar políticas de grupo (GPOs) en Windows Server 2022 utilizando filtrado avanzado por seguridad, para controlar con precisión qué usuarios o equipos reciben configuraciones específicas.

  • Reestructuración de Active Directory

    Antes de aplicar GPOs con filtrado avanzado, se reestructuró el entorno de Active Directory. Se crearon nuevas OUs por departamento (VentasDept, RRHHDept, etc.) para mejorar la segmentación de políticas. Esta decisión se tomó tras detectar conflictos en el filtrado dentro de una OU única con múltiples grupos.


  • Creación de GPO personalizado

    Se creó un nuevo objeto de directiva de grupo (GPO) llamado Bloqueo_Panel_deControl desde la consola Administración de directivas de grupo. Este GPO será utilizado para restringir el acceso al Panel de Control en usuarios específicos mediante filtrado avanzado.


  • Edición de GPO para bloqueo del Panel de Control

    Se editó la GPO Bloqueo_Panel_deControl en la ruta Configuración de Usuario → Plantillas administrativas → Panel de control, habilitando la opción Prohibir el acceso a Configuración de PC y al Panel de control. Esta configuración impide que los usuarios de Dept_Ventas accedan a herramientas de configuración del sistema, reforzando la seguridad y el control administrativo.


  • Verificación de aplicación de GPO

    Se verificó la aplicación de la GPO Bloqueo_Panel_deControl mediante dos métodos:

    • Visual:

      El usuario Rodolfo Pineda intentó abrir el Panel de Control y recibió un mensaje indicando que una política impide la acción.


    • Técnico:

      El comando gpresult /R confirmó que la GPO está aplicada correctamente al usuario.


Resultado y aprendizaje obtenido:
Se logró crear una estructura de OUs segmentada por departamento, vinculando la GPO Bloqueo_Panel_deControl exclusivamente a Dept_Ventas mediante configuración de usuario. Se verificó su aplicación efectiva mediante prueba visual y técnica, confirmando que los usuarios afectados no pueden acceder al Panel de Control, sin impactar a usuarios técnicos como IT. Esta práctica refuerza el uso estratégico de GPOs, la importancia de una estructura lógica en Active Directory y la validación precisa de políticas aplicadas.

Objetivo: Aplicar medidas de endurecimiento (hardening) en un entorno Windows Server para mejorar la seguridad base del sistema.

  • Revisión y desinstalación de roles innecesarios

    Se accedió al asistente Quitar roles de servidor desde el Administrador del servidor en Windows Server 2022, ejecutándose en Oracle VirtualBox. En la sección Roles de servidor, se validó que la mayoría de los roles innecesarios ya estaban deshabilitados, incluyendo servicios como Fax Server, Print Services y Remote Desktop Services.

    Solo permanecen activos los roles esenciales para la operación de red local, como Servidor DHCP y Servidor DNS. Esta configuración inicial contribuye a reducir la superficie de ataque del sistema y optimiza el rendimiento del servidor.


  • Creación de GPO para políticas de contraseña segura

    Se creó la directiva Seguridad_Contraseña desde Group Policy Management, aplicándola a nivel de dominio. En la ruta Configuración del equipo > Configuración de Windows > Configuración de seguridad > Directivas de cuenta > Directiva de contraseñas, se establecieron los siguientes parámetros:

    • Aplicar el historial de contraseñas: 24 contraseñas recordadas
    • Longitud mínima de la contraseña: 8 caracteres (recomendado: 12)
    • Requisitos de complejidad: Activado
    • Duración máxima de la contraseña: 30 días (recomendado: 60)
    Estas configuraciones fortalecen la autenticación de usuarios, reduciendo el riesgo de accesos no autorizados por contraseñas débiles o repetidas.


  • Asignación de acceso remoto a usuarios técnicos

    Desde Usuarios y equipos de Active Directory, se accedió a las propiedades del usuario Josue Sosa ubicado en la unidad organizativa Dept_IT. En la pestaña Miembro de, se verificó que el usuario pertenece al grupo Usuarios del escritorio remoto, lo que le permite iniciar sesión por RDP sin privilegios administrativos.

    Esta asignación forma parte de la estrategia de endurecimiento, limitando el acceso remoto únicamente a usuarios técnicos autorizados. Se recomienda complementar esta configuración con una GPO que restrinja el inicio de sesión remoto a este grupo.


  • Configuración de auditoría básica de eventos

    Se creó la directiva Auditoria_Seguridad y se vinculó a la unidad organizativa Dept_IT. Desde el editor de directivas, se accedió a la ruta Configuración del equipo → Configuración de Windows → Configuración de seguridad → Directivas locales → Directiva de auditoría, donde se habilitaron las siguientes políticas con auditoría de Éxito y Error:

    • Auditar eventos de inicio de sesión
    • Auditar eventos de administración de cuentas
    • Auditar el acceso a objetos
    Estas configuraciones permiten registrar intentos de acceso, modificaciones de cuentas y actividades sobre archivos protegidos.


  • Restricción de uso de dispositivos USB mediante GPO

    Se creó la directiva Restriccion_USB y se vinculó a la unidad organizativa Dept_IT. Desde el editor de directivas, se accedió a la ruta Configuración del equipo > Plantillas administrativas > Sistema > Acceso al almacenamiento extraíble, donde se revisaron las políticas disponibles para restringir el uso de dispositivos USB, discos externos y otros medios extraíbles.

    Posteriormente, se habilitó la política Todas las clases de almacenamiento extraíble: Denegar acceso a todos, bloqueando el uso de estos dispositivos en los equipos del departamento técnico. Esta medida refuerza la seguridad contra fugas de información y malware.


  • Bloqueo del Panel de Control y Configuración para usuarios no técnicos

    Se creó la directiva Bloqueo_PanelControl y se vinculó a la unidad organizativa Dept_RRHH, que contiene usuarios no técnicos. Desde el editor de directivas, se accedió a la ruta Configuración del usuario > Plantillas administrativas > Panel de control, habilitando la política Prohibir el acceso al Panel de control y a la configuración de PC.

    Esta medida impide que los usuarios realicen cambios en configuraciones del sistema, mejorando el control administrativo y reduciendo riesgos operativos. Se recomienda aplicar filtrado por grupo si la OU contiene usuarios mixtos.

    🛈 Nota: Esta directiva fue previamente aplicada y documentada en la sección Aplicación de GPOs con filtrado avanzado, donde se utilizó filtrado por grupo para restringir el acceso al Panel de Control en usuarios no técnicos.


  • Revisión y fortalecimiento de cuentas locales

    Nota: Este paso fue omitido en el servidor actual por tratarse de un controlador de dominio. La revisión de cuentas se realiza desde Usuarios y equipos de Active Directory, validando pertenencia a grupos, privilegios y estado de las cuentas técnicas.


  • Validación de actualizaciones del sistema

    Se detectaron actualizaciones pendientes críticas en el servidor, incluyendo parches de seguridad y componentes de .NET Framework. Se procedió a instalar las actualizaciones desde el módulo de Configuración, seguido de un reinicio para aplicar los cambios.


    • El sistema quedó actualizado y protegido contra vulnerabilidades conocidas.

Registro Técnico

Bitácora de eventos, configuraciones y soluciones aplicadas durante prácticas técnicas.

La estructura inicial del dominio empresa.local contenía una única OU UsuariosEmpresa con usuarios de distintos departamentos agrupados por grupos de seguridad. Esta configuración generó conflictos al aplicar GPOs con filtrado avanzado.

Nota: Aunque las OUs aparecen debajo del contenedor Users en la vista alfabética, están ubicadas directamente en la raíz del dominio, lo que permite vincular GPOs de forma más eficiente.

Decidí reestructurar las OUs para mejorar la administración, auditoría y segmentación de políticas. Cada departamento ahora cuenta con su propia OU, lo que facilita la aplicación de GPOs específicas.

La GPOs Bloqueo_Panel_deControl, Restriccion_USB y Seguridad_Contraseña fueron vinculadas tanto a Dept_Ventas como a Dept_RRHH. Esta estrategia permite mantener una configuración centralizada y reducir la cantidad de objetos de directiva en el dominio.
